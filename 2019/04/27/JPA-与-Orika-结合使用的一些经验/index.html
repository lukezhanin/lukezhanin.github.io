JPA 与 Orika 结合使用的一些经验

发表于 2019-04-27 |

Background

项目中经常会遇到需要在 DTO 和 Entity 之间拷贝属性的场景。一个个手工去复制既麻烦也容易出错，自然开源界有很多工具可以帮助我们来实现这个需求，比如Spring 的 BeanUtils、 Orika BeanMapper、MapStruct、Dozer 等。

从内部原理来说，可以分为反射派和字节码生成派。反射派历来为人所诟病的是性能，字节码则会多占用一些内存。两全其美的办法也有，那就是手写，可是手写繁琐且扩展性差，下次加了一个属性忘改对应的代码就完蛋了。世间事就是如此，哪有“不负如来不负卿”的好事呢？

从扩展性和性能综合考虑，最后选择了字节码生成派的 Orika BeanMapper。

Pitfalls

工厂使用

第一版时，每次 map 时都去注册一下 classMap，结果运行了几天之后，内存暴涨，一看才发现是 Orika 的类缓存占用，谨记MapperFactory 必须使用单例。

集合

默认情况下，JPA 使用的实现是 Hibernate。如果设置一对多关系，实体中的集合类使用的是代理类 PersitentBag，而经过 Orika 映射的集合却是单纯的集合，于是保存到数据库时，Hibernate 会抛异常，不允许保存。

Bean 注册

Orika 本身支持自动映射，也就是说如果没有显示注册 Bean 类型，那么会自动生成代理类，但与 Hibernate 使用时，会碰到各种莫名其妙的代理问题，因此需要手工注册 Bean。如果每个需要使用到 BeanMapper 的类都要手动注册 Bean 的映射，那也太麻烦了。

需要忽略的字段

同理，有些字段我们不想要暴露到外部接口，或者从 DTO 中更新，于是在映射的时候要告诉 Orika 忽略掉。
